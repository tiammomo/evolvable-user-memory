# 隐私生命周期设计

> **状态：设计与验收基线，尚未实现。**
>
> 当前版本没有同意/处理依据登记、保留策略执行、抑制 API、删除编排或删除证明。本文定义未来实现必须满足的语义和测试标准，不能作为当前产品具备隐私合规能力的声明。具体法定义务、保留期限和例外必须由部署方的法律、安全与数据治理负责人确认。

## 1. 目标与边界

隐私生命周期必须覆盖一条数据从“允许摄入”到“停止使用、清理并可证明”的全过程，同时保持项目的五平面边界：

- Evidence 在保留期间只追加，不因后续信念而改写；
- Belief 修正追加不可变版本，不静默覆盖历史；
- Experience 只从可归因 Outcome 学习；
- Projection 可丢弃、可重建，也必须可停止使用和清理；
- Evolution 不能修改同意、保留、抑制、删除或审计规则。

“不可变”表示数据在获准保留期间不能被静默改写，不表示数据必须永久存在。经过授权的生命周期操作可以删除完整实体或去标识化内容，但必须留下不含原始内容的最小证明。

本文不决定特定司法辖区的合法处理依据，也不定义 UI 文案、法定响应时限或哪些数据必须依法保留。

## 2. 数据清单与责任

实现前必须建立可机器读取的数据清单。至少覆盖：

| 数据类别 | 当前或未来位置 | 主要风险 | 生命周期要求 |
| --- | --- | --- | --- |
| Observation 与 EvidenceSpan | PostgreSQL 权威状态或内存模式 | 原始输入可能高度敏感 | 摄入前检查、最短保留、默认不记录日志、可清理 |
| MemoryRecord 与 MemoryRevision | PostgreSQL 权威状态或内存模式 | 推断出的偏好仍可能是个人数据 | Scope/目的限制、即时抑制、版本链整体处理 |
| RecallTrace 与 Trace item | PostgreSQL 权威状态 | 查询、上下文和返回内容可泄露意图 | 独立保留期、访问限制、删除/去标识策略 |
| Outcome 与 note | PostgreSQL 权威状态 | 业务结果及自由文本可能敏感 | 可归因、最小化 note、独立保留策略 |
| UtilityEstimate | PostgreSQL 权威状态 | 从行为推断出的效用画像 | 随源 Revision 与 Scope 清理或重新计算 |
| outbox event | PostgreSQL | 含 Scope、标识和事件元数据 | 不含原始证据正文、受控消费、随生命周期清理 |
| embedding/graph/摘要投影 | 尚未实现 | 难以逐项定位、可能跨 Scope 泄漏 | 可定位、可重建、可验证清理、游标可追踪 |
| 前端状态与缓存 | 浏览器内存；未来可能增加缓存 | Scope 切换后的残留 | Scope 切换立即清理，不持久化敏感正文 |
| 日志、指标、追踪 | 部署环境 | 二次复制和长期泄漏 | 默认无原始证据、明确保留期和访问控制 |
| 备份与恢复副本 | 部署环境 | 在线删除后仍可恢复 | 过期策略、恢复后重放删除屏障、可审计演练 |

每增加一个存储、缓存、索引、分析管道或导出功能，都必须先登记数据类别、负责人、用途、保留期、Scope 键、删除处理器和验证方式。未登记的副本不得进入生产。

## 3. 建议的策略模型

以下对象均为待实现设计：

- `ProcessingGrant`：Scope、允许用途、处理依据、有效期、策略版本和签发主体；
- `RetentionPolicy`：按数据类别定义保留上限、触发时间和合法例外；
- `SuppressionFence`：阻止某 Scope、记录或用途继续写入、召回和投影；
- `ErasureRequest`：幂等请求、可信发起者、目标范围、原因、状态和期限；
- `ErasureReceipt`：只保存策略版本、请求 ID、Scope 摘要、时间、计数和各处理器状态，不保存原始内容；
- `LifecycleHandler`：对一个已登记存储执行抑制、删除或去标识，并返回可验证结果。

这些策略必须由受信任的治理控制面管理，通过类型化 application port 注入。API 请求体、模型输出、Outcome 或策略演化都不能自行授予处理权限或延长保留期。

## 4. 建议的状态与转换

```text
摄入前评估
  ├─ 不允许 ──> REJECTED（不写入证据正文）
  └─ 允许 ────> ACTIVE
                    │ 撤回、用户请求、风险处置或到期
                    ▼
                SUPPRESSED
                    │ 删除请求或保留期到期
                    ▼
             ERASURE_PENDING
                    │ 所有登记处理器完成或记录允许的保留例外
                    ▼
                  ERASED
```

转换规则：

1. `SUPPRESSED` 和 `ERASURE_PENDING` 必须立即从召回、列表、投影和新写入路径中失败关闭，即使物理清理尚未完成。
2. 清理失败不能恢复为 `ACTIVE`；它保持 `ERASURE_PENDING` 并告警、重试。
3. `ERASED` 只保留最小证明。后续新输入必须重新完成摄入前评估，不能“恢复”旧内容。
4. 任何保留例外都必须由固定治理策略授权，记录范围、期限和审批主体；演化模块无权创建例外。
5. 状态转换必须是 Scope 内幂等操作，并能抵抗写入、召回和投影消费者的并发竞态。

## 5. 生命周期工作流

### 5.1 摄入前检查

在创建 Observation 之前，服务端从可信身份派生 Scope，并根据用途检查 `ProcessingGrant`。没有明确授权、策略不可用或策略已过期时失败关闭。拒绝记录只能包含最小审计元数据，不能复制被拒绝的证据正文。

### 5.2 修正与透明度

修正继续追加 Revision，保留 `supersedes_revision_id`。面向用户的导出必须区分原始证据、当前信念、历史信念和结果效用，不能把推断当成用户原话。导出本身也必须鉴权、按 Scope 限制并审计。

### 5.3 抑制

抑制是“立即停止使用”，不是物理删除的同义词。抑制屏障应在权威查询之前生效，并传播给未来投影、缓存和消费者。被抑制数据不得召回、列表、训练、评估或产生新投影；只有经过授权的隐私处理器可以读取其最小定位信息。

### 5.4 保留期到期

保留任务按权威 UTC 时间和版本化策略计算到期，不依赖客户端时钟。到期时先建立抑制屏障，再创建幂等 ErasureRequest。扫描器重复运行不能重复删除、重复发布或延长期限。

### 5.5 删除编排

删除编排至少按以下顺序执行：

1. 在权威存储原子建立 Scope/目标抑制屏障和幂等 ErasureRequest；
2. 拒绝并发的新写入和召回，等待在途事务到达一致性边界；
3. 清理 Observation、EvidenceSpan、Revision、Trace、Outcome、Utility，以及未来任何与目标 Scope 关联的策略选择记录；全局 StrategySnapshot 不得因单个 subject 被误删；
4. 发布不含原始内容的生命周期事件，清理所有已登记投影、缓存和导出副本；
5. 处理 outbox 中仍含个人标识的旧事件，禁止普通重放复活已删除数据；
6. 记录备份到期时间，并保证任何备份恢复后优先重放抑制/删除屏障；
7. 所有处理器完成后生成最小 ErasureReceipt。

物理级联删除、加密擦除或去标识化必须按数据类别选择并经过威胁评估。不能只删除当前 Revision 而保留可还原内容的 Evidence、Trace、Outcome、投影或自由文本日志。

### 5.6 删除证明

证明应回答“哪项请求、按哪个策略、哪些数据类别、何时、由哪些处理器完成或被合法保留”，而不是保存被删除数据。证明中的 Scope 应使用受控摘要或内部请求标识；不得包含证据正文、记忆 value、查询、Outcome note 或可逆导出。

证明状态必须区分：`pending`、`partially_completed`、`completed`、`exception_recorded` 和 `failed`。只要任何必需处理器未确认，就不能向调用方声称删除已经完成。

## 6. 与 outbox、投影和备份的关系

当前只实现了 PostgreSQL 同事务 outbox 写入，尚无消费者或独立投影。未来实现必须满足：

- 普通领域事件与生命周期事件按每个 Scope/aggregate 有可验证顺序；
- 消费者幂等，重复投递不会恢复被抑制或删除的数据；
- 投影记录源 Revision、Scope 和游标，能精确定位与确定性重建；
- 删除屏障优先于积压的创建/更新事件；
- 死信、重放和人工修复都需要授权、审计和范围限制；
- 备份不能承诺即时逐块删除，但必须有明确最长存续期，并验证恢复后不会重新暴露已删除数据。

## 7. 实现验收标准

以下条件全部满足前，不能把隐私生命周期标记为“已实现”：

### 身份与策略

- [ ] Scope 只从可信认证/授权上下文派生，伪造 payload Scope 无效。
- [ ] 没有处理依据、策略服务不可用或授权过期时，摄入失败关闭且不保存原始正文。
- [ ] 处理依据绑定用途；一个用途的授权不能被另一个用途复用。
- [ ] 演化、模型输出和普通管理员不能修改治理策略或创建保留例外。

### 隔离与并发

- [ ] 抑制、到期和删除请求均按 tenant/subject/目标隔离，并有跨 Scope 负向测试。
- [ ] 重复请求返回同一逻辑结果，不重复生成副作用或证明。
- [ ] 写入与删除、召回与抑制、投影消费与删除之间有竞态测试；任何时序都不能让已建立屏障的数据重新可见。
- [ ] 删除处理中依赖故障时保持 `ERASURE_PENDING`，召回继续失败关闭。

### 数据覆盖

- [ ] 数据清单中的每个权威表、outbox、投影、缓存、日志类别、导出和备份都有处理器、负责人和验证证据。
- [ ] Trace、Outcome、Utility 和自由文本 note 的处理策略被单独测试，而不只测试当前记忆列表。
- [ ] outbox 与指标不含原始证据正文；日志测试能检测意外正文泄漏。
- [ ] 恢复旧备份后，删除屏障在对外服务前重放，已删除数据不可查询或重新投影。

### 证明与运维

- [ ] ErasureReceipt 不含原始内容或可逆个人数据，并能验证策略版本、覆盖范围和处理器结果。
- [ ] 未完成、部分完成和保留例外不会被呈现为 `completed`。
- [ ] 保留扫描、删除积压、最老待处理请求、处理器失败和备份到期有指标与告警。
- [ ] 故障注入、恢复演练和抽样核验形成可追溯但不含敏感正文的审计证据。

## 8. 实现前必须决定的问题

- 哪些处理依据和用途由产品支持，谁有权签发或撤回？
- Observation、Revision、Trace、Outcome 和审计证明各自的默认/最大保留期是什么？
- 哪些法定例外允许保留哪些最小字段，最长多久？
- 删除粒度是整个 subject、某类记忆、单条 record，还是它们的组合？
- 多区域、导出、分析系统和备份的最长清理时限是什么？
- 删除证明由谁签名、保存多久，如何避免证明本身成为新的身份数据？

这些决定需要形成版本化政策和自动化测试，不能只存在于运维手册或人工约定中。
